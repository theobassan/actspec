# v0.2 — JavaScript / Node actions

> **Light stub.** Detailed module spec + a sandbox spike land **once v0.1 is validated** (depth policy — [specs/README](../README.md#depth-policy)). This is the pointer, not the contract.

**Goal:** test `using: node20` / `node24` / future Node actions through the **same** `mock` / `run` / `expect` surface. The test author never learns it's a JS action.

## Adds
- **`@actharness/node`** — the node `ActionExecutor` + `JsSandbox` + the network mock layer.
- `@actharness/coverage` gains the **`jsLines`** layer (V8 coverage → Istanbul, via c8).
- **No public API change** — `mockGitHubApi`/`mockNetwork` already exist in [API.md §2](../../docs/API.md).

## The genuinely hard parts (why a spike comes first)
- **`JsSandbox` transparency** — the core bet is "wire the protocol env-files and real `@actions/core` just works." Validate against 3–4 real published actions *before* committing the design ([ARCHITECTURE → highest-risk assumptions](../../docs/ARCHITECTURE.md#the-three-highest-risk-assumptions)). Watch for: ESM vs CJS entrypoints, bundled (`ncc`) actions, `process.exit`, `process.env` isolation per `worker_thread`.
- **pre/main/post lifecycle + `GITHUB_STATE`** — JS actions have three phases with state threaded between them ([Fidelity](../../docs/ARCHITECTURE.md#fidelity--semantics)); already modeled as `StepResult.phase` in v0.1's types.
- **Internal-dependency mocking** — `mockGitHubApi` (undici `MockAgent` for Octokit) and `mockNetwork`, mounted into the sandbox. Distinct from `uses:` mocking; same "mock your dependencies" model.

## Coverage gained
`+ jsLines` (real line coverage of the action's JS).

## Acceptance (high level)
A real-shaped node action: reads inputs, calls `core.setOutput`/`core.setSecret`, optionally hits the GitHub API → `run()` returns its outputs from `$GITHUB_OUTPUT`; `mockGitHubApi` intercepts Octokit; a masked secret never appears in `annotations`; `pre`/`post` phases run and are assertable.

All new packages meet the [CONVENTIONS Definition of Done](../../docs/CONVENTIONS.md#definition-of-done-every-module) — including 100% coverage (`pnpm test:coverage`): statements, branches, functions, and lines.

## Spike status

The sandbox spike is **complete** — see [specs/spikes/node-sandbox-findings.md](../spikes/node-sandbox-findings.md). H1–H2, H4–H7 validated; H3 (ESM entrypoints) is confirmed architecturally supported but not explicitly exercised. **Before shipping v0.2:** add an ESM fixture action (`.mjs` or `"type":"module"` package) and confirm `await import()` in the worker handles it correctly. This is the first step of v0.2 implementation, not a separate spike.

## Implementation note

The node-sandbox spike in `spike/node-sandbox/` produced a working `JsSandbox` with `worker_threads`, protocol env-file wiring, and `process.exit` handling. **Do not rewrite from scratch** — copy and evolve that code into `@actharness/node`. The spike code is the starting point; the module spec is the finishing gate.
